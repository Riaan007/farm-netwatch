Open Kuma ↗

Run it with docker compose --profile kuma up -d, create the Kuma admin account, and enter it above. Then open a device and tick "Monitor in Uptime Kuma" — Netwatch builds that monitor (tagged with its category) and pushes its status each scan. Only devices you tick are added.

Bulk-create Kuma ping monitors (tagged by category). Each monitor auto-follows its device's IP, so it keeps working when DHCP moves the device. Already-monitored devices are skipped.

Devices behind these MACs are tracked by IP so each one shows separately. Once you fix the bridge/WiFi device itself (e.g. disable proxy-ARP), remove its MAC here to go back to normal MAC tracking — the devices re-merge (or appear under their real MACs) on the next scan.

Managed extra IPs (multi-subnet)

Adds a second IP so the Pi sits on another LAN and ARP-scans it. Additive — never touches your main connection. Re-applied on reboot.

DHCP ↔ Fixed IP

⚠️ A wrong fixed IP/gateway can cut this Pi off. Changes auto-revert to DHCP unless you confirm — on a remote site, confirm only after you've checked the dashboard still loads.

Fixed-IP control needs the NetworkManager image (docker-compose.netcfg.yml) — not enabled on this host.

Everything needed to rebuild this site on a new Pi: settings, device names/categories, saved logins and the hub-VPN link. The hub also backs this up automatically every day. Treat the file like a password.

Remove offline devices not seen in days

Forgets stale/offline devices (gear from another network, or kit you've retired) from the list, registry and uptime history. Use 0 days to remove all currently-offline devices. Anything still live reappears on the next scan.
